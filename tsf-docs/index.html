Multi-Tenant Isolation

Every request is scoped to a tenant via TenantMiddleware. The middleware extracts X-Tenant-Id from the request header and stores it in thread-local storage.

Tenant Resolution Order

  1. X-Tenant-Id header (set by frontend)
  2. User's organization_id (fallback)
  3. SaaS super-org (for federation admin)

Data Model

class TenantModel(models.Model):
    organization = models.ForeignKey(Organization, on_delete=CASCADE)
    class Meta:
        abstract = True

class VerifiableModel(TenantModel):
    lifecycle_status = CharField(choices=LIFECYCLE_STATUS)
    locked_by = ForeignKey(User)
    locked_at = DateTimeField()
    current_verification_level = IntegerField(default=0)
    class Meta:
        abstract = True

Transaction Lifecycle Engine

Universal, configurable multi-level verification pipeline for all transactional models. Any model inheriting VerifiableModel automatically gains lifecycle support.

Status Flow

OPEN
→
LOCKED
→
VERIFIED
→
CONFIRMED
  • OPEN — Editable. User can add/edit/remove lines.
  • LOCKED — Frozen for review. Requires comment to unlock.
  • VERIFIED — Multi-level verification chain (1→2→3...).
  • CONFIRMED — Final. Ready for posting/execution.

Lifecycle Service

TransactionLifecycleService.lock(instance, transaction_type, user)
TransactionLifecycleService.unlock(instance, transaction_type, user, comment)    # mandatory
TransactionLifecycleService.verify(instance, transaction_type, user)             # auto-confirms at final level
TransactionLifecycleService.unverify(instance, transaction_type, user, comment)  # mandatory
TransactionLifecycleService.seed_defaults(organization)
TransactionLifecycleService.get_history(transaction_type, id)

Per-organization, per-transaction-type config stored in TransactionVerificationConfig.

Audit Trail

Every action is logged to TransactionStatusLog with who, what, when, why, and IP address. Immutable.

TransactionStatusLog:
    transaction_type, transaction_id    # polymorphic link
    action          (LOCK|UNLOCK|VERIFY|UNVERIFY|CONFIRM)
    level           (1, 2, 3...)
    performed_by    → User
    performed_at    (auto timestamp)
    comment         (mandatory for UNLOCK, UNVERIFY)
    ip_address

ViewSet Mixin

Any ViewSet can gain lifecycle endpoints by adding LifecycleViewSetMixin:

class MyOrderViewSet(LifecycleViewSetMixin, TenantModelViewSet):
    lifecycle_transaction_type = 'MY_TYPE'
    # Automatically adds: lock/, unlock/, verify/, unverify/, lifecycle_history/

Finance & Accounting

Chart of Accounts

Tree-structured accounts with numeric codes. Two levels:

  • Ledger (Synthetic) — Header accounts, aggregate children, no direct postings
  • Financial (Analytical) — Leaf accounts, receive journal entry line items
ChartOfAccount:
    code (e.g. "1110"), name, type (ASSET|LIABILITY|EQUITY|REVENUE|EXPENSE)
    level (LEDGER|FINANCIAL), parent, is_active
    
Types: 1xxx=Assets, 2xxx=Liabilities, 3xxx=Equity, 4xxx=Revenue, 5xxx=Expenses

Journal Entries

Double-entry bookkeeping with journal entry lines that must balance (debits = credits).

JournalEntry → JournalEntryLine[]
    account, debit, credit, description
    Status: DRAFT → POSTED
    Scope: OFFICIAL | INTERNAL

Vouchers

Source documents that generate journal entries. Full CRUD with auto-reference via TransactionSequence.

FieldDescription
referenceAuto-generated (e.g. VOU-000001)
voucher_typePAYMENT_IN, PAYMENT_OUT, RECEIPT, etc.
accountFinancial account involved
journal_entryGenerated JE when posted
statusDRAFT → POSTED → CANCELLED

Direct Expenses

Immediate one-time expenses with automatic GL posting.

DirectExpense:
    name, category (RENT|UTILITIES|SALARIES|...), amount, date
    source_account → FinancialAccount
    expense_coa → ChartOfAccount
    
Post action creates:
    1. FinancialEvent (EXPENSE type)
    2. JournalEntry (Dr: expense COA, Cr: source account COA)
    3. Links event + JE back to expense

Deferred Expenses

Expenses amortized over time (insurance, prepaid rent). Recognition happens monthly.

Assets & Depreciation

Capital assets with straight-line depreciation schedules. Auto-generates amortization entries.

Loan Contracts

Loan management with installment schedules, payment tracking, and interest calculation.

Financial Events

Audit trail of every financial action: sales, purchases, adjustments, transfers, payments.

SaaS Architecture

Federation-level admin panel for managing multiple organizations. The first organization acts as the super-admin.

Organizations

Each org is a fully isolated tenant with its own data, modules, and settings. Managed via /saas/organizations.

Available on any ViewSet using LifecycleViewSetMixin:
